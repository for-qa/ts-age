Variables & Scope

let vs const vs var

✓

Analogy: var = a whiteboard anyone can erase (old style, leaks everywhere). let = a notebook page you can update but only inside the room you're in. const = a carved stone tablet — permanent, can't be overwritten.

Featurevarletconst
ScopeFunctionBlock { }Block { }
Reassign✅✅❌
Redeclare✅❌❌
Hoisted✅ (undefined)❌ (TDZ)❌ (TDZ)
Recommended❌ Never✅ When value changes✅ Default choice
 
var x = 1; var x = 2;     // ✅ No error (bad!)
let y = 1; // let y = 2;  // ❌ Cannot redeclare
const z = 1; // z = 2;    // ❌ Cannot reassign

// Block scope difference
if (true) { var a = 10; }
console.log(a);  // 10 (var leaks out! ❌)

if (true) { let b = 10; }
// console.log(b); // ❌ ReferenceError (let stays in block ✅)
Variables & Scope

Block Scope vs Local (Function) Scope

✓

Analogy: Block scope = a locker that only exists inside a specific classroom — gone when you leave the room (the { }). Local scope = a personal bag you carry for the entire length of a function call — alive until the function returns.

FeatureBlock ScopeLocal / Function Scope
Created byAny { } — if, for, while, bare blocksA function body
Keywordslet, constvar, let, const, function params
LifetimeOnly while inside the { }Until the function returns
Accessible outside?❌ No❌ No (but wider than block)
Example containersif, for, while, switchAny named or anonymous function
var behaviour⚠ var IGNORES block scopevar respects function scope
 
// ─── BLOCK SCOPE (let / const) ───────────────────────────
function demo() {
  if (true) {
    let blockVar = "I live only in this if-block";
    const blockConst = 42;
    console.log(blockVar);   // ✅ works here
  }
  // console.log(blockVar);  // ❌ ReferenceError — out of block!
}

// ─── var LEAKS out of blocks (but NOT out of functions) ──
function leakyDemo() {
  if (true) {
    var leaked = "I escape the block!";
  }
  console.log(leaked); // ✅ "I escape the block!" — bad practice!
}
// console.log(leaked); // ❌ ReferenceError — var IS bound to function

// ─── LOCAL (FUNCTION) SCOPE ──────────────────────────────
function calcTax(price: number) {
  const rate = 0.18;          // local to calcTax — not visible outside
  return price * (1 + rate);
}
// console.log(rate); // ❌ ReferenceError — local scope!

// ─── Nested scopes — inner can see outer, not vice versa ─
function outer() {
  let x = 10;                 // outer scope
  function inner() {
    let y = 20;               // inner (block/local) scope
    console.log(x + y);      // ✅ 30 — inner sees outer
  }
  inner();
  // console.log(y);          // ❌ outer cannot see inner's y
}
⚠

Key rule: Always prefer let/const so your variables respect block boundaries. var only respects function scope, which makes bugs much harder to track down in loops and conditionals.

Hoisted vs Non-Hoisted

✓

Analogy: Hoisted = a student who raises their hand before the teacher even asks — JS moves the declaration to the top automatically. Non-hoisted = a student who waits to be called — you must declare it first or you get an error.

FeatureHoistedNon-Hoisted
Moved to top?✅ Declaration moved to top❌ Stays where written
Use before declare?✅ (but value is undefined)❌ ReferenceError (TDZ)
Variablesvarlet, const
Functionsfunction declarationsArrow functions, function expressions
Classes❌ Classes are NEVER hoisted
 
// ✅ HOISTED — var declaration is moved to the top
console.log(x);  // undefined (not an error!)
var x = 10;
console.log(x);  // 10

// ❌ NON-HOISTED — let/const stay in place (TDZ)
// console.log(y);  // ✓ ReferenceError: Cannot access 'y' before initialization
let y = 20;
console.log(y);  // 20

// ✅ Function declarations ARE hoisted
greet();  // "Hello!" — works even before definition!
function greet() { console.log("Hello!"); }

// ❌ Arrow functions are NOT hoisted
// sayBye();  // ✓ ReferenceError
const sayBye = () => console.log("Bye!");
⚠

TDZ (Temporal Dead Zone): The space between the start of a block and where let/const is declared. Accessing the variable in the TDZ causes a ReferenceError — this is by design to catch bugs early.

== vs === (Loose vs Strict)

✓

Analogy: == = "Do these look similar?" (a photo vs a painting of the same person — close enough). === = "Are these the EXACT same thing?" (same person, same fingerprint, same ID).

Feature== (Loose)=== (Strict)
Type conversion✅ Yes (coerces)❌ No
5 == "5"truefalse
0 == falsetruefalse
null == undefinedtruefalse
Recommended❌ Avoid✅ Always use
 
console.log(5 == "5");    // true  (string → number coercion)
console.log(5 === "5");   // false (different types — no coercion)
console.log(0 == false);  // true  (false → 0)
console.log(0 === false); // false (number ≠ boolean)
console.log("" == false); // true  (both → 0)
console.log("" === false);// false

Primitive vs Non-Primitive

✓

Analogy: Primitive = a single coin (one value, copied directly). Non-primitive = a wallet (holds many things, you share the wallet itself — not a copy).

FeaturePrimitiveNon-Primitive
StoresSingle valueCollection / complex structure
Typesstring, number, boolean, null, undefined, bigint, symbolobject, array, function, class, interface
Stored inStack (fast)Heap (reference)
ComparisonBy valueBy reference
Mutable?❌ Immutable✅ Mutable
 
// Primitive — copied by VALUE
let a = 10;
let b = a;
b = 20;
console.log(a); // 10 (unchanged — b got its own copy)

// Non-Primitive — copied by REFERENCE
let arr1 = [1, 2, 3];
let arr2 = arr1;       // both point to the SAME array in memory
arr2.push(4);
console.log(arr1);     // [1, 2, 3, 4] (changed! same reference)

Immutable vs Mutable

✓

Analogy: Immutable = a printed book — once printed, you can't change the words. You have to print a new edition. Mutable = a whiteboard — erase and rewrite anytime.

FeatureImmutableMutable
Can be changed?❌ No — creates a new value✅ Yes — modifies in place
Examplesstring, number, boolean, tuple (TS)array, object
Methods returnNew value (original unchanged)Modified original
Keywordsconst, readonly, Readonly<T>Default behavior for objects/arrays
Safety✅ Predictable, no side effects⚠ Can cause unexpected changes
 
// IMMUTABLE — strings cannot be changed in place
let name = "hello";
let upper = name.toUpperCase();  // returns NEW string
console.log(name);   // "hello"  (original unchanged!)
console.log(upper);  // "HELLO"  (new value)

// MUTABLE — arrays CAN be changed in place
let nums = [1, 2, 3];
nums.push(4);         // modifies the ORIGINAL array
console.log(nums);    // [1, 2, 3, 4] (changed!)

// Making things immutable with readonly (TypeScript)
const config: Readonly<{ port: number; host: string }> = {
  port: 3000,
  host: "localhost"
};
// config.port = 8080;  // ❌ ERROR — readonly!

// Immutable array
const colors: readonly string[] = ["red", "green"];
// colors.push("blue");  // ❌ ERROR — readonly array!

null vs undefined

✓

Analogy: undefined = an empty parking spot where nobody has parked yet. null = a parking spot with a "Reserved — intentionally kept empty" sign.

Featureundefinednull
MeaningNot yet assignedIntentionally empty
Set byJavaScript automaticallyDeveloper manually
typeof"undefined""object" (JS bug!)
== each other?✅ null == undefined is true

let x;                        // undefined (never assigned)

console.log(x == y);   // true  (loose equality — both "empty")

// Nullish coalescing ?? — handles both null and undefined

any vs unknown

✓

Analogy: any = a wild card — does whatever it wants, no checks. unknown = a new employee who must show their ID and pass verification before accessing anything.

Featureanyunknown

Access properties✅ No checks needed❌ Must narrow first

// any — no safety (dangerous!)
let a: any = "hello";

// unknown — must check first (safe!)
let b: unknown = "hello";

Data Types

never vs void

✓

Analogy: void = a function that finishes its job but doesn't hand you anything back (like a postman who delivers but gives you no receipt). never = a function that never comes back at all — it crashes, loops forever, or throws before it can return.

Featurevoidnever
MeaningReturns, but no useful valueNever returns at all
Value allowedundefined onlyNothing — no value possible
Used forFunctions with no return valueThrow-only or infinite-loop functions
Assignable toOnly void / undefinedEvery type (bottom type)
Exhaustiveness check❌✅ Great for switch guards
 
// void — function finishes, returns nothing
function logMessage(msg: string): void {
  console.log(msg);
  // no return needed
}

// never — function NEVER reaches a return
function crash(msg: string): never {
  throw new Error(msg);  // always throws — never returns
}

function infiniteLoop(): never {
  while (true) { }      // never returns
}

// never for exhaustive switch checks
type Shape = "circle" | "square";
function area(s: Shape): number {
  switch (s) {
    case "circle":  return Math.PI;
    case "square":  return 1;
    default:
      const check: never = s;  // ✅ TS errors if a case is missing!
      return check;
  }
}

Union vs Tuple

✓

Analogy: Union = "this gift can be a book OR a toy" (one slot, many possible types). Tuple = "this box must contain exactly: 1 book in slot 0, 1 pen in slot 1, 1 eraser in slot 2" — fixed positions, fixed types.

FeatureUnionTuple
What it definesONE value with multiple possible typesFIXED array with specific type per position
Syntaxstring | number[string, number]
LengthN/A (single value)Fixed
AccessNarrow first, then useBy index — type known immediately
 
// Union — ONE slot, multiple possible types
let id: string | number = "ABC";
id = 123;  // ✅ also valid

// Tuple — FIXED structure: position 0=string, 1=number, 2=boolean
let record: [string, number, boolean] = ["Alice", 25, true];
console.log(record[0]); // "Alice" (TypeScript knows it's a string)
console.log(record[1]); // 25     (TypeScript knows it's a number)

// Common use: React useState returns a tuple!
// const [count, setCount] = useState<number>(0);
// count is number, setCount is Dispatch<SetStateAction<number>>

Enum vs Union Literal

✓

Analogy: Enum = a physical menu card on the table (exists at runtime, you can pick it up). Union literal = the waiter reads options aloud (only exists at order time — no card remains).

FeatureEnumUnion Literal
Runtime value✅ Exists in compiled JS❌ Erased after compile
Reverse mapping✅ (numeric enums)❌
Bundle sizeLarger (generates JS object)Zero (compile-time only)
Iterate values✅❌
Preferred todaySpecial cases only✅ Modern TS preferred
 
// Enum — generates a real JS object at runtime
enum Direction { Up, Down, Left, Right }
console.log(Direction.Up);      // 0
console.log(Direction[0]);      // "Up" (reverse mapping!)
let d = Direction.Up;

// Union literal — zero runtime cost, better inference
type Direction2 = "up" | "down" | "left" | "right";
let d2: Direction2 = "up";

// const enum — inline, even smaller bundle (no reverse mapping)
const enum Status { Active = "active", Inactive = "inactive" }
let s: Status = Status.Active;  // compiles to just "active"

Literal Types vs Generics

✓

Analogy: Literal = a vending machine with 3 fixed buttons (Coke, Pepsi, Water — nothing else). Generic = a universal dispenser — you load ANY drink and it adapts.

FeatureLiteral TypesGenerics <T>
ValuesFixed, known at design timeFlexible, decided at call time
FlexibilityRestricted to exact valuesWorks with ANY type
Use caseConfig options, status codes, modesReusable utilities, containers, hooks
 
// Literal — ONLY these exact string values allowed
type Theme = "light" | "dark" | "system";
let t: Theme = "light";   // ✅
// let t2: Theme = "blue"; // ❌ not in the literal set

// Generic — works with ANY type — decided at call site
function wrap<T>(val: T): T[] { return [val]; }
wrap("hello");   // inferred as string[]
wrap(42);        // inferred as number[]
wrap({ id: 1 }); // inferred as { id: number }[]

// Real-world: Array<T>, Promise<T>, Ref<T> are all generics!

Type Assertion vs Type Narrowing

✓

Analogy: Type assertion = telling the bouncer "Trust me, I'm 18" without showing ID — you override the check, which can be wrong. Type narrowing = actually showing your ID — the bouncer (TypeScript) verifies it themselves.

FeatureType Assertion (as)Type Narrowing
How it worksYou tell TS what the type isTS figures it out from runtime checks
Safety⚠ Unsafe — you could be wrong✅ Safe — proven at runtime
Syntaxvalue as Typetypeof, instanceof, in, truthiness
Runtime check?❌ Compile-time only✅ Actual runtime check
PreferredOnly when you're certain✅ Always prefer this
 
// TYPE ASSERTION — you override TS (risky!)
const input = document.getElementById("name") as HTMLInputElement;
input.value = "Alice";  // TS trusts you — but if it's not an input, runtime crash!

// Double assertion (worst case — code smell!)
const bad = "hello" as unknown as number;  // ❌ Avoid this pattern

// TYPE NARROWING — TS verifies it (safe!)
function printId(id: string | number) {
  if (typeof id === "string") {
    console.log(id.toUpperCase());  // ✅ TS knows it's string here
  } else {
    console.log(id.toFixed(2));     // ✅ TS knows it's number here
  }
}

// instanceof narrowing
function logError(e: unknown) {
  if (e instanceof Error) {
    console.log(e.message);  // ✅ narrowed to Error
  }
}

// 'in' narrowing
type Cat = { meow: () => void };
type Dog = { bark: () => void };
function makeSound(animal: Cat | Dog) {
  if ("meow" in animal) {
    animal.meow();  // ✅ narrowed to Cat
  } else {
    animal.bark();  // ✅ narrowed to Dog
  }
}

function vs Arrow Function

✓

Analogy: Regular function = a formal letter writer who uses their own "I" (has its own this). Arrow function = a sticky note writer who borrows the "I" from whoever owns the desk (inherits parent's this).

FeaturefunctionArrow =>
Syntaxfunction name() {}const name = () => {}
this bindingOwn this (dynamic)Inherits parent's this (lexical)
Hoisted✅ Yes❌ No
As class method✅ Preferred⚠ Careful with this
CallbacksWorks fine✅ Preferred (shorter, correct this)
 
// Regular — hoisted, own 'this'
function greet(name: string) { return `Hi ${name}`; }

// Arrow — shorter, inherits parent 'this'
const greet2 = (name: string) => `Hi ${name}`;

// 'this' difference matters critically in classes
class Timer {
  seconds = 0;
  start() {
    // ✅ Arrow inherits Timer's 'this'
    setInterval(() => { this.seconds++; }, 1000);

    // ❌ Regular function gets its own 'this' (undefined in strict mode!)
    // setInterval(function() { this.seconds++; }, 1000);
  }
}

for...in vs for...of

✓

Analogy: for...in = reading the INDEX labels on library shelves (shelf 0, shelf 1...). for...of = reading the actual BOOKS sitting on those shelves.

Featurefor...infor...of
Iterates overKeys / property names / indexesValues
ArraysReturns index as string ("0","1"...)Returns actual value ✅
Objects✅ Returns keys❌ Objects not iterable by default
StringsReturns index✅ Returns each character

let fruits = ["apple", "banana", "cherry"];

for (let i in fruits) console.log(i);        // "0", "1", "2"  (string indexes!)
for (let f of fruits) console.log(f);        // "apple", "banana", "cherry" (values!)

let obj = { a: 1, b: 2, c: 3 };
for (let key in obj) console.log(key);       // "a", "b", "c"
// for (let val of obj) { }                  // ❌ ERROR — plain objects not iterable

// Strings work with for...of
for (let ch of "hi") console.log(ch);        // "h", "i"

Optional vs Default Parameters

✓

Analogy: Optional parameter = a form field marked "optional" — you may leave it blank (value is undefined if skipped). Default parameter = a pre-filled form field — it has a sensible value already, but you can override it.

FeatureOptional (param?)Default (param = val)
Syntaxname?: stringname: string = "Guest"
Value if skippedundefinedThe default value
Must handle undefined?✅ Always narrow it❌ Already has a value
PositionMust come after required paramsMust come after required params
Use caseTruly optional configSensible fallbacks
 
// Optional — might be undefined inside the function
function greet(name?: string): string {
  // Must handle the undefined case!
  return `Hello, ${name ?? "stranger"}!`;
}
greet("Alice");  // "Hello, Alice!"
greet();         // "Hello, stranger!"

// Default — always has a value, simpler to use
function greet2(name: string = "Guest"): string {
  return `Hello, ${name}!`;  // no undefined check needed
}
greet2("Bob");   // "Hello, Bob!"
greet2();        // "Hello, Guest!"

// Mixing — required, then default, then optional
function createUser(
  id: number,                   // required
  role: string = "viewer",      // default
  nickname?: string             // optional
) {
  console.log(id, role, nickname);
}
createUser(1);                  // 1, "viewer", undefined
createUser(2, "admin");         // 2, "admin",  undefined
createUser(3, "admin", "Alex"); // 3, "admin",  "Alex"

Class vs Interface

✓

Analogy: Class = a factory that builds cars — it has the blueprint AND the machines to produce them. Interface = just the blueprint on paper — no machines, someone else builds it.

extends vs implements

✓

Analogy: extends = inheriting your parent's house — you GET the furniture already inside. implements = signing a contract that says you'll furnish your own house — you PROMISE to provide everything yourself.

Featureextendsimplements

public vs private vs protected

✓

Analogy: public = the front porch — anyone can visit. private = your bedroom — only you (the class itself) can enter. protected = the family room — only you and your family members (subclasses) can enter.

Featurepublicprivateprotected
Accessible fromAnywhereOwn class onlyOwn class + subclasses
Default?✅ Yes❌❌
Subclass can access?✅❌✅
Outside instance?✅❌ TS compile error❌ TS compile error
JS runtime privacyN/ATS only (use # for true JS private)TS only
 
class BankAccount {
  public  owner: string;      // anyone can read/write
  private balance: number;    // ONLY this class can touch it
  protected accountNo: string; // this class + subclasses

  constructor(owner: string, balance: number, accountNo: string) {
    this.owner     = owner;
    this.balance   = balance;
    this.accountNo = accountNo;
  }

  public deposit(amount: number) {
    this.balance += amount;   // ✅ private accessible inside class
  }

  public getBalance() {
    return this.balance;      // ✅ exposing private via public method (encapsulation!)
  }
}

class SavingsAccount extends BankAccount {
  showAccount() {
    console.log(this.accountNo);  // ✅ protected OK in subclass
    // console.log(this.balance); // ❌ private — not accessible in subclass!
  }
}

const acc = new BankAccount("Alice", 1000, "ACC123");
console.log(acc.owner);         // ✅ public
// console.log(acc.balance);    // ❌ TS Error — private!
// console.log(acc.accountNo);  // ❌ TS Error — protected!

// True JS private with # (ES2022)
class Secret {
  #pin: number = 1234;   // real JS private — not just TS!
  getPin() { return this.#pin; }
}
Async

Synchronous vs Asynchronous

✓

Analogy: Sync = a bank queue — each person waits for the one in front before getting served. Async = a restaurant — you order food, chat while the kitchen cooks, and food arrives when ready without you waiting frozen at the table.

FeatureSynchronousAsynchronous
ExecutionLine by line, in orderNon-blocking, runs in background
Blocking✅ Waits for each operation❌ Continues executing other code
Speed for I/OSlow (CPU waits idle)Fast (CPU does other work)
KeywordsNormal functionsasync, await, Promise
Use casesMath, string ops, array transformsAPI calls, file I/O, timers, DB queries
 
// Sync — blocks until done, order guaranteed
console.log("1");
console.log("2");
console.log("3");  // Output: 1, 2, 3

// Async — doesn't block
console.log("1");
setTimeout(() => console.log("2"), 1000);
console.log("3");  // Output: 1, 3, 2  (2 comes last!)
Async

Promise vs async / await

✓

Analogy: Promise + .then() = a callback chain — you hand the baton to the next runner at each step. async/await = you write the relay race as if all runners are YOU, one after the other — same async underneath, but reads like sync code.

FeaturePromise (.then/.catch)async / await
Syntax styleChain of callbacksReads like synchronous code
Error handling.catch()try / catch
Readability⚠ Chaining can get messy✅ Clean and linear
Parallel executionPromise.all([])await Promise.all([])
Under the hoodPromisesPromises (syntactic sugar!)
DebuggingHarder (stack traces fragmented)✅ Easier (natural call stack)
 
// ── PROMISE (.then / .catch) ──────────────────────────────
function fetchUser(id: number): Promise<string> {
  return fetch(`/api/users/${id}`)
    .then(res => res.json())
    .then(data => data.name)
    .catch(err => { console.error(err); return "Unknown"; });
}

// ── ASYNC / AWAIT (same result, cleaner!) ─────────────────
async function fetchUser2(id: number): Promise<string> {
  try {
    const res  = await fetch(`/api/users/${id}`);
    const data = await res.json();
    return data.name;
  } catch (err) {
    console.error(err);
    return "Unknown";
  }
}

// ── PARALLEL — run multiple at once (don't await in sequence!) ──
async function fetchAll() {
  // ❌ Sequential (slow) — waits one by one
  const a = await fetchUser2(1);
  const b = await fetchUser2(2);

  // ✅ Parallel (fast) — both fire simultaneously
  const [c, d] = await Promise.all([fetchUser2(1), fetchUser2(2)]);
}
⚠

Common mistake: Calling await in a loop one by one is sequential — use Promise.all() to run them in parallel and get a significant speed boost.
