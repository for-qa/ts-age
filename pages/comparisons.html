Data Types

Primitive vs Non-Primitive

✓

Analogy: Primitive = a single coin (one value). Non-primitive = a wallet (holds many things inside).

FeaturePrimitiveNon-Primitive
StoresSingle valueCollection / complex structure
Typesstring, number, boolean, null, undefined, bigint, symbolobject, array, function, class, interface
Stored inStack (fast)Heap (reference)
ComparisonBy valueBy reference
Mutable?❌ Immutable✅ Mutable
 
// Primitive — copied by VALUE
let a = 10;
let b = a;
b = 20;
console.log(a); // 10 (unchanged!)

// Non-Primitive — copied by REFERENCE
let arr1 = [1, 2, 3];
let arr2 = arr1;
arr2.push(4);
console.log(arr1); // [1, 2, 3, 4] (changed! same reference)
Data Types

Immutable vs Mutable

✓

Analogy: Immutable = a printed book — once printed, you can't change the words on the page. You can only buy a new edition (create a new value). Mutable = a whiteboard — you can erase and rewrite content anytime.

FeatureImmutableMutable
Can be changed?❌ No — creates a new value✅ Yes — modifies in place
Examplesstring, number, boolean, tuple (TS)array, object
Methods returnNew value (original unchanged)Modified original
Keywordsconst, readonly, Readonly<T>Default behavior for objects/arrays
Safety✅ Predictable, no side effects⚠ Can cause unexpected changes
 
// IMMUTABLE — strings cannot be changed in place
let name = "hello";
let upper = name.toUpperCase();  // returns NEW string
console.log(name);   // "hello"  (original unchanged!)
console.log(upper);  // "HELLO"  (new value)

// MUTABLE — arrays CAN be changed in place
let nums = [1, 2, 3];
nums.push(4);         // modifies the ORIGINAL array
console.log(nums);    // [1, 2, 3, 4] (changed!)

// Making things immutable with readonly
const config: Readonly<{ port: number; host: string }> = {
  port: 3000,
  host: "localhost"
};
// config.port = 8080;  // ❌ ERROR — readonly!

// Immutable array
const colors: readonly string[] = ["red", "green"];
// colors.push("blue");  // ❌ ERROR — readonly array!

null vs undefined

✓

Analogy: undefined = an empty parking spot (nobody parked yet). null = a parking spot with a "Reserved" sign (intentionally empty).

Featureundefinednull
MeaningNot yet assignedIntentionally empty
Set byJavaScript automaticallyDeveloper manually
typeof"undefined""object" (JS bug!)
== each other?✅ null == undefined is true
=== each other?❌ null === undefined is false
 
let x;                 // undefined (never assigned)
let y: string | null = null;  // null (intentionally empty)

console.log(x == y);   // true  (loose equality)
console.log(x === y);  // false (strict — different types)
Data Types

any vs unknown

✓

Analogy: any = a wild kid who does whatever they want (no rules). unknown = a new employee who must show ID before accessing anything (rules enforced).

Featureanyunknown
Type safety❌ None✅ Full
Access properties✅ No checks needed❌ Must narrow first
Assign to other types✅ Freely❌ Must check first
Use caseQuick prototypingProduction code
 
// any — no safety (dangerous!)
let a: any = "hello";
a.toFixed(2);  // No error at compile, CRASHES at runtime! ✓

// unknown — must check first (safe!)
let b: unknown = "hello";
// b.toUpperCase();  // ❌ ERROR — must narrow!
if (typeof b === "string") {
  b.toUpperCase();   // ✅ Safe after check
}
Variables

let vs const vs var

✓

Analogy: var = a whiteboard anyone can erase (old style). let = a notebook page you can update. const = a carved stone tablet (permanent).

Featurevarletconst
ScopeFunctionBlock { }Block { }
Reassign✅✅❌
Redeclare✅❌❌
Hoisted✅ (undefined)❌ (TDZ)❌ (TDZ)
Recommended❌ Never✅ When value changes✅ Default choice
 
var x = 1; var x = 2;     // ✅ No error (bad!)
let y = 1; // let y = 2;  // ❌ Cannot redeclare
const z = 1; // z = 2;    // ❌ Cannot reassign

// Block scope difference
if (true) { var a = 10; }
console.log(a);  // 10 (var leaks out!)

if (true) { let b = 10; }
// console.log(b); // ❌ ERROR (let stays in block)

== vs === (Loose vs Strict)

✓

Analogy: == = "Do these look similar?" (a photo vs painting of same person). === = "Are these the EXACT same thing?" (same person, same ID card).

Feature== (Loose)=== (Strict)
Type conversion✅ Yes (coerces)❌ No
5 == "5"truefalse
0 == falsetruefalse
null == undefinedtruefalse
Recommended❌ Avoid✅ Always use
 
console.log(5 == "5");    // true  (string → number)
console.log(5 === "5");   // false (different types!)
console.log(0 == false);  // true  (false → 0)
console.log(0 === false); // false (number ≠ boolean)
Variables

Hoisted vs Non-Hoisted

✓

Analogy: Hoisted = a student who raises their hand before the teacher asks — JavaScript moves the declaration to the top automatically. Non-hoisted = a student who waits to be called — you must declare it before using it.

FeatureHoistedNon-Hoisted
Moved to top?✅ Declaration moved to top❌ Stays where written
Use before declare?✅ (but value is undefined)❌ ReferenceError (TDZ)
Variablesvarlet, const
Functionsfunction declarationsArrow functions, function expressions
Classes❌ Classes are NEVER hoisted
 
// ✅ HOISTED — var declaration is moved to the top
console.log(x);  // undefined (not an error!)
var x = 10;
console.log(x);  // 10

// ❌ NON-HOISTED — let/const stay in place (TDZ)
// console.log(y);  // ✓ ReferenceError: Cannot access 'y' before initialization
let y = 20;
console.log(y);  // 20

// ✅ Function declarations ARE hoisted
greet();  // "Hello!" — works even before definition!
function greet() { console.log("Hello!"); }

// ❌ Arrow functions are NOT hoisted
// sayBye();  // ✓ ReferenceError
const sayBye = () => console.log("Bye!");
⚠

TDZ (Temporal Dead Zone): The space between the start of a block and where let/const is declared. Accessing the variable in the TDZ causes a ReferenceError — this is by design to catch bugs early.

type vs interface

✓

Analogy: interface = a formal contract (can be extended). type = a nickname (can be anything — unions, primitives, combos).

Featuretypeinterface
Object shapes✅✅
Primitives✅❌
Unions A | B✅❌
Intersections A & B✅❌
extends❌ (use &)✅
implements✅✅
Declaration merging❌✅
 
// type — flexible (unions, primitives)
type ID = string | number;
type Point = { x: number; y: number };

// interface — extendable (OOP contracts)
interface Animal { name: string; }
interface Dog extends Animal { breed: string; }
Types

Union vs Tuple

✓

Analogy: Union = "this gift can be a book OR a toy" (one value, many possible types). Tuple = "this box has exactly: 1 book, 1 pen, 1 eraser" (fixed positions, fixed types).

FeatureUnionTuple
What it definesONE value, multiple possible typesFIXED array with specific types per position
Syntaxstring | number[string, number]
LengthN/A (single value)Fixed
AccessNarrow then useBy index
 
// Union — ONE value can be string OR number
let id: string | number = "ABC";
id = 123;  // ✅ also valid

// Tuple — FIXED structure: [string, number, boolean]
let record: [string, number, boolean] = ["Alice", 25, true];
console.log(record[0]); // "Alice" (string)
console.log(record[1]); // 25 (number)
Types

Enum vs Union Literal

✓

Analogy: Enum = a physical menu card on the table (exists at runtime). Union literal = the waiter reads options aloud (exists only at order time, no card left).

FeatureEnumUnion Literal
Runtime value✅ Exists in JS❌ Erased after compile
Reverse mapping✅ (numeric)❌
Bundle sizeLargerZero
Iterate values✅❌
 
// Enum — generates runtime JS object
enum Direction { Up, Down, Left, Right }
let d = Direction.Up;  // 0

// Union literal — zero runtime cost
type Direction2 = "up" | "down" | "left" | "right";
let d2: Direction2 = "up";
Types

Literal Types vs Generics

✓

Analogy: Literal = a vending machine with 3 fixed buttons (Coke, Pepsi, Water). Generic = a universal dispenser — you load ANY drink brand and it adapts.

FeatureLiteral TypesGenerics <T>
ValuesFixed, known at design timeFlexible, decided at call time
FlexibilityRestricted to exact valuesWorks with ANY type
Use caseConfig options, status codesReusable functions, containers
 
// Literal — ONLY these exact values
type Theme = "light" | "dark";
let t: Theme = "light";   // ✅
// let t2: Theme = "blue"; // ❌

// Generic — works with ANY type
function wrap<T>(val: T): T[] { return [val]; }
wrap("hello");  // string[]
wrap(42);       // number[]
Functions

function vs Arrow Function

✓

Analogy: Regular function = a formal letter (has its own "I"/this). Arrow function = a sticky note (borrows the "I"/this from its parent).

FeaturefunctionArrow =>
Syntaxfunction name() {}const name = () => {}
this bindingOwn thisInherits parent's this
Hoisted✅ Yes❌ No
As method✅ Preferred⚠ Careful with this
CallbacksWorks✅ Preferred (shorter)
 
// Regular — hoisted, own 'this'
function greet(name: string) { return `Hi ${name}`; }

// Arrow — shorter, inherits 'this'
const greet2 = (name: string) => `Hi ${name}`;

// 'this' difference matters in classes
class Timer {
  seconds = 0;
  start() {
    setInterval(() => { this.seconds++; }, 1000);  // ✅ arrow inherits 'this'
    // setInterval(function() { this.seconds++; }, 1000); // ❌ 'this' is wrong!
  }
}
Functions

for...in vs for...of

✓

Analogy: for...in = reading the INDEX labels on library shelves (shelf 0, shelf 1...). for...of = reading the actual BOOKS on those shelves.

Featurefor...infor...of
Iterates overKeys / indexesValues
ArraysReturns index (string!)Returns value
Objects✅ Returns keys❌ Not iterable
Recommended for arrays❌✅
 
let fruits = ["apple", "banana", "cherry"];

for (let i in fruits) console.log(i);      // "0", "1", "2" (keys!)
for (let f of fruits) console.log(f);      // "apple", "banana", "cherry" (values!)

let obj = { a: 1, b: 2 };
for (let key in obj) console.log(key);     // "a", "b"
// for (let val of obj) { } // ❌ ERROR — objects not iterable!
OOP

Class vs Interface

✓

Analogy: Class = a factory that builds cars (has blueprints AND machines). Interface = just the blueprint (no machines — someone else builds it).

FeatureClassInterface
Implementation✅ Full❌ Abstract only
Constructor✅❌
new keyword✅❌
Access modifiers✅❌
Multiple inheritance❌ Single only✅ Multiple
Runtime existence✅ In JS❌ Erased
 
// Interface — structure only
interface Shape { area(): number; }

// Class — full implementation
class Circle implements Shape {
  constructor(public radius: number) {}
  area() { return Math.PI * this.radius ** 2; }
}
OOP

extends vs implements

✓

Analogy: extends = inheriting your parent's house (you GET everything). implements = signing a contract (you PROMISE to provide everything yourself).

Featureextendsimplements
Used withClass → Class, Interface → InterfaceClass → Interface
PropertiesAuto-inheritedMust re-declare
MethodsAuto-inherited (can override)Must implement
Multiple❌ Single only✅ Multiple OK
 
// extends — auto-inherits everything
class Animal { move() { console.log("moving"); } }
class Dog extends Animal { }  // Dog already has move()!

// implements — must provide everything
interface Flyable { fly(): void; }
class Bird implements Flyable {
  fly() { console.log("flying"); }  // MUST implement!
}
Async

Synchronous vs Asynchronous

✓

Analogy: Sync = bank queue (wait for each person). Async = restaurant (order food, chat while kitchen cooks, food arrives later).

FeatureSynchronousAsynchronous
ExecutionLine by line, in orderNon-blocking, background
Blocking✅ Waits for each line❌ Continues executing
Speed for I/OSlow (waits idle)Fast (does other work)
KeywordsNormal functionsasync, await, Promise
Use casesMath, string opsAPI calls, file I/O, timers
 
// Sync — blocks until done
console.log("1");
console.log("2");
console.log("3");  // Output: 1, 2, 3

// Async — doesn't block
console.log("1");
setTimeout(() => console.log("2"), 1000);
console.log("3");  // Output: 1, 3, 2
