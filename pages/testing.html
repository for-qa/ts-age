Why Write Tests?

✓

Real-Life: Tests are like spell-check for code ✓. Every time you change something, tests automatically verify nothing else broke. Without tests, you're proofreading a 500-page book by eye every time you change a word.

!
Catch bugs before users do
Why
// Without tests:
// "I changed function A... did it break B, C, and D?"
// → Manually check everything? No way.

// With tests:
// Run `npm test` → ✅ 47 passed, 0 failed
// Confidence that nothing broke!

// Tools: Jest (most popular) or Vitest (faster, Vite-based)
// Both have nearly identical syntax
Basics

describe / it / expect

1
The three building blocks
Structure
// Function to test
function add(a: number, b: number): number {
  return a + b;
}

// Tests:
describe("add", () => {               // Group of tests
  it("should add two positive numbers", () => {
    expect(add(2, 3)).toBe(5);         // Assertion
  });

  it("should handle negative numbers", () => {
    expect(add(-1, -2)).toBe(-3);
  });

  it("should handle zero", () => {
    expect(add(0, 5)).toBe(5);
  });
});

// Read like English:
// "Describe 'add': it should add two positive numbers"
Basics

Common Matchers

2
Different ways to assert
Matchers
// EQUALITY
expect(2 + 2).toBe(4);              // exact (===)
expect({ a: 1 }).toEqual({ a: 1 }); // deep equality (objects)
// toBe = same reference, toEqual = same content

// TRUTHINESS
expect(true).toBeTruthy();
expect(null).toBeFalsy();
expect(null).toBeNull();
expect(undefined).toBeUndefined();
expect("hello").toBeDefined();

// NUMBERS
expect(10).toBeGreaterThan(5);
expect(10).toBeLessThanOrEqual(10);
expect(0.1 + 0.2).toBeCloseTo(0.3);  // floating point!

// STRINGS
expect("hello world").toContain("world");
expect("hello").toMatch(/^hel/);

// ARRAYS
expect([1, 2, 3]).toContain(2);
expect([1, 2, 3]).toHaveLength(3);

// ERRORS
expect(() => { throw new Error("boom"); }).toThrow("boom");
expect(() => JSON.parse("invalid")).toThrow();

// NEGATION — .not
expect(5).not.toBe(3);
expect([1, 2]).not.toContain(5);
Patterns

Setup / Teardown

3
Run code before/after each test
Lifecycle
describe("UserService", () => {
  let users: string[];

  // Runs BEFORE each test — fresh state every time
  beforeEach(() => {
    users = ["Alice", "Bob"];
  });

  // Runs AFTER each test — cleanup
  afterEach(() => {
    users = [];
  });

  it("should add a user", () => {
    users.push("Charlie");
    expect(users).toHaveLength(3);
  });

  it("should start with 2 users (fresh state!)", () => {
    // beforeEach ran again — users reset to ["Alice", "Bob"]
    expect(users).toHaveLength(2);
  });
});

// Also: beforeAll / afterAll — run once for entire describe block
Patterns

Mocking

4
Fake dependencies for isolation
Mocks
// Mock function — track calls and set return values
const mockFn = jest.fn();
mockFn("hello");
expect(mockFn).toHaveBeenCalledWith("hello");
expect(mockFn).toHaveBeenCalledTimes(1);

// Mock with return value
const getUser = jest.fn().mockReturnValue({ name: "Alice" });
expect(getUser()).toEqual({ name: "Alice" });

// Mock async function
const fetchData = jest.fn().mockResolvedValue({ id: 1 });
const data = await fetchData();
expect(data).toEqual({ id: 1 });

// WHY mock?
// Don't call REAL APIs in tests — they're slow and unreliable
// Mock the API function, test YOUR logic around it
Patterns

Testing Async Code

5
await in tests works just like regular code
Async
// Async function to test
async function fetchUser(id: number): Promise<{ name: string }> {
  const res = await fetch(`/api/users/${id}`);
  return res.json();
}

// Test with async/await
it("should fetch user", async () => {
  const user = await fetchUser(1);
  expect(user.name).toBe("Alice");
});

// Test that async function throws
it("should throw on invalid id", async () => {
  await expect(fetchUser(-1)).rejects.toThrow("Not found");
});
Watch Out!

⚠ Common Mistakes

⚠
Mistake 1: Testing implementation, not behavior 
// ❌ Testing HOW it works (fragile — breaks when refactored)
expect(calculator.internalSum).toBe(5);

// ✅ Testing WHAT it does (stable)
expect(calculator.getTotal()).toBe(5);
⚠
Mistake 2: Tests sharing state 
// ❌ Test 2 depends on test 1's data
// Tests should be INDEPENDENT — use beforeEach to reset
⚠
Mistake 3: toBe vs toEqual for objects 
// ❌ toBe checks reference, not content!
expect({ a: 1 }).toBe({ a: 1 });    // FAILS!

// ✅ toEqual checks deep equality
expect({ a: 1 }).toEqual({ a: 1 }); // PASSES ✅

Cheatsheet

// STRUCTURE
describe("group", () => {
  beforeEach(() => { /* setup */ });
  it("should do X", () => { expect(X).toBe(Y); });
});

// MATCHERS
.toBe(val)        .toEqual(obj)     .toBeTruthy()
.toBeNull()       .toBeDefined()    .toContain(item)
.toThrow()        .toMatch(/regex/) .toHaveLength(n)
.not.toBe(val)    .toBeCloseTo(n)

// MOCKS
jest.fn()                          // create mock
jest.fn().mockReturnValue(x)       // set return
jest.fn().mockResolvedValue(x)     // async return
expect(fn).toHaveBeenCalledWith(x) // verify call

// ASYNC
it("async", async () => { await expect(fn()).resolves.toBe(x); });
