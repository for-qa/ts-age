Core Concept

Why Error Handling?

✓

Real-Life Example: Error handling is like a seatbelt in a car. You hope you never need it, but when something goes wrong (a crash/bug), it saves you from total disaster. Without it, one small problem can crash your entire application.

Errors happen — network fails, users enter bad data, files don't exist. Error handling lets your code deal with failures gracefully instead of crashing.

// ❌ Without error handling — app CRASHES
let data = JSON.parse("invalid json");  // ✓ SyntaxError!
console.log("This line never runs");    // never reached

// ✅ With error handling — app RECOVERS
try {
  let data = JSON.parse("invalid json");
} catch (error) {
  console.log("Bad JSON, using default data");
}
console.log("App continues running!");  // ✅ This runs!

Core Syntax

try / catch / finally

✓

Real-Life Example: try = attempting a cooking recipe. catch = the fire extinguisher if something burns. finally = cleaning the kitchen regardless of whether the dish was good or burnt.

try {
  // Code that MIGHT throw an error
  let result = riskyOperation();
  console.log(result);
} catch (error) {
  // Runs ONLY if an error occurs in try block
  console.log("Something went wrong:", error);
} finally {
  // Runs ALWAYS — whether error occurred or not
  console.log("Cleanup complete");
}
BlockWhen it runsRequired?
tryAlways — contains risky code✅ Yes
catchOnly when an error occursNeed at least one of catch/finally
finallyAlways — regardless of errorNeed at least one of catch/finally
 
// Practical example
function divide(a: number, b: number): number {
  try {
    if (b === 0) {
      throw new Error("Cannot divide by zero!");
    }
    return a / b;
  } catch (error) {
    console.log(`Error: ${error}`);
    return 0;  // fallback value
  } finally {
    console.log(`Attempted: ${a} / ${b}`);
  }
}

console.log(divide(10, 2));   // 5 (success)
console.log(divide(10, 0));   // 0 (caught error, returned fallback)

Throwing

throw — Create Your Own Errors

✓

Real-Life Example: Like a quality inspector on a factory line. If a product doesn't meet standards, the inspector THROWS it off the line (rejects it). Your code can do the same — throw errors when data doesn't meet your requirements.

function withdraw(balance: number, amount: number): number {
  if (amount <= 0) {
    throw new Error("Amount must be positive!");
  }
  if (amount > balance) {
    throw new Error("Insufficient funds!");
  }
  return balance - amount;
}

try {
  console.log(withdraw(1000, 500));   // 500 ✅
  console.log(withdraw(1000, 2000));  // throws! ❌
} catch (error) {
  if (error instanceof Error) {
    console.log(error.message);  // "Insufficient funds!"
  }
}
// Validate function input
function setAge(age: number): void {
  if (age < 0 || age > 150) {
    throw new RangeError(`Invalid age: ${age}. Must be 0-150.`);
  }
  console.log(`Age set to ${age}`);
}

setAge(25);   // ✅ "Age set to 25"
setAge(-5);   // ❌ RangeError: Invalid age: -5
Custom Errors

Custom Error Classes

✓

Real-Life Example: Like different types of hospital alerts. A "Code Blue" (cardiac) is different from a "Code Red" (fire). Custom errors let you create specific error types so you can handle each one differently.

// Custom error class — extends built-in Error
class ValidationError extends Error {
  field: string;

  constructor(field: string, message: string) {
    super(message);        // call parent Error constructor
    this.name = "ValidationError";
    this.field = field;
  }
}

class NotFoundError extends Error {
  resource: string;

  constructor(resource: string) {
    super(`${resource} not found`);
    this.name = "NotFoundError";
    this.resource = resource;
  }
}

// Handle different error types differently
function processUser(id: number): void {
  try {
    if (id <= 0) throw new ValidationError("id", "ID must be positive");
    if (id > 1000) throw new NotFoundError("User");
    console.log(`Processing user ${id}`);
  } catch (error) {
    if (error instanceof ValidationError) {
      console.log(`Invalid ${error.field}: ${error.message}`);
    } else if (error instanceof NotFoundError) {
      console.log(`Missing: ${error.resource}`);
    } else {
      console.log(`Unexpected: ${error}`);
    }
  }
}

processUser(5);      // "Processing user 5"
processUser(-1);     // "Invalid id: ID must be positive"
processUser(9999);   // "Missing: User"
TypeScript-Specific

unknown vs any in catch blocks

In TypeScript, the catch parameter is unknown by default (since TS 4.4). This is safer than any.

// ❌ Unsafe — 'any' lets you do anything without checks
try { /* ... */ } catch (error: any) {
  console.log(error.message);  // No error, but risky!
  // What if 'error' is a string? It won't have .message!
}

// ✅ Safe — 'unknown' forces you to check first
try { /* ... */ } catch (error: unknown) {
  // error.message;  // ❌ ERROR — must narrow first!

  if (error instanceof Error) {
    console.log(error.message);  // ✅ Safe — checked!
  } else if (typeof error === "string") {
    console.log(error);           // ✅ Safe — it's a string
  } else {
    console.log("Unknown error");
  }
}
ApproachType SafetyUsage
catch (error: any)❌ NoneQuick scripts only
catch (error: unknown)✅ FullProduction code (recommended)
catch (error)Depends on TS configDefault behavior
TypeScript-Specific

Error Type Guard — Reusable Helper

// Reusable type guard for errors
function isError(value: unknown): value is Error {
  return value instanceof Error;
}

// Even better — handle non-Error throws
function getErrorMessage(error: unknown): string {
  if (error instanceof Error) return error.message;
  if (typeof error === "string") return error;
  return "An unknown error occurred";
}

// Clean usage everywhere
try {
  throw "Something went wrong";  // string thrown (not Error!)
} catch (error) {
  console.log(getErrorMessage(error));  // "Something went wrong"
}

try {
  throw new Error("Database down");
} catch (error) {
  console.log(getErrorMessage(error));  // "Database down"
}
Async

Async Error Handling

// async/await + try/catch
async function fetchData(url: string): Promise<any> {
  try {
    const response = await fetch(url);

    if (!response.ok) {
      throw new Error(`HTTP Error: ${response.status}`);
    }

    return await response.json();
  } catch (error) {
    if (error instanceof Error) {
      console.log(`Fetch failed: ${error.message}`);
    }
    return null;  // fallback
  }
}

// Promise .catch()
fetch("/api/users")
  .then(res => res.json())
  .catch(err => console.log(`Error: ${err}`));
i

Tip: For async functions, try/catch (with await) and .catch() (with promises) are two sides of the same coin. Use whichever reads cleaner for your use case.

Advanced

Result Pattern — Type-Safe Alternative

✓

Real-Life Example: Like a medical test result. Instead of the doctor suddenly fainting (throwing an error), they calmly hand you a report: "Result: Positive" or "Result: Negative." The result is always returned — you decide how to handle it.

// Define a Result type
type Result<T> =
  | { success: true; data: T }
  | { success: false; error: string };

// Function returns Result instead of throwing
function parseAge(input: string): Result<number> {
  const age = parseInt(input);

  if (isNaN(age)) {
    return { success: false, error: `"${input}" is not a number` };
  }
  if (age < 0 || age > 150) {
    return { success: false, error: `Age ${age} out of range` };
  }

  return { success: true, data: age };
}

// Usage — no try/catch needed!
const result = parseAge("25");

if (result.success) {
  console.log(`Age: ${result.data}`);    // TS knows data exists
} else {
  console.log(`Error: ${result.error}`); // TS knows error exists
}

parseAge("abc");   // { success: false, error: '"abc" is not a number' }
parseAge("200");   // { success: false, error: 'Age 200 out of range' }
parseAge("30");    // { success: true, data: 30 }
Guidelines

Best Practices

#PracticeWhy
1Use unknown not any in catchForces type checking — safer
2Always check instanceof ErrorNot all thrown values are Error objects
3Create custom error classesHandle different failures differently
4Use finally for cleanupClose connections, release resources
5Don't catch and ignoreSilent failures are the hardest to debug
6Throw early, catch lateValidate inputs early, handle errors at the top level
 
// ❌ DON'T — catch and ignore
try { riskyOp(); } catch (e) { }  // silent failure! ✓

// ✅ DO — at minimum, log the error
try { riskyOp(); } catch (e) {
  console.error("Operation failed:", e);
}

// ❌ DON'T — catch too broadly
try {
  // 100 lines of code
} catch (e) { /* which line failed? */ }

// ✅ DO — wrap only the risky part
const data = prepareData();  // not risky
try {
  await sendToServer(data);  // risky!
} catch (e) {
  console.error("Send failed:", e);
}
