Utility Types

Pre-built type transformations that ship with TypeScript. Instead of rewriting types from scratch, transform existing ones with a single word.

What

What are Utility Types?

✓

Real-Life: Utility types are like LEGO modification kits. You already have a model (your type). Instead of rebuilding it, you use a kit to make all pieces removable (Partial), glue them permanently (Readonly), or remove specific pieces (Omit).

!
Transform types without rewriting them
Core Idea
interface User {
  name: string;
  age: number;
  email: string;
}

// Instead of writing a whole new type for updates:
// interface UserUpdate { name?: string; age?: number; email?: string; }

// Just use Partial:
type UserUpdate = Partial<User>;
// Same result! All properties become optional
Object Types

Partial<T> — Make All Optional

1
Every property becomes optional (?)
Partial
interface User {
  name: string;
  age: number;
  email: string;
}

// Partial<User> = { name?: string; age?: number; email?: string; }

// ✅ Real use: update functions
function updateUser(id: number, updates: Partial<User>) {
  // Can pass any subset of User fields
}

updateUser(1, { name: "Alice" });           // ✅ just name
updateUser(1, { age: 26, email: "a@b" });   // ✅ age + email
updateUser(1, {});                           // ✅ nothing
Object Types

Required<T> — Make All Required

2
Opposite of Partial — removes all ?
Required
interface Config {
  host?: string;
  port?: number;
  debug?: boolean;
}

// Required<Config> = { host: string; port: number; debug: boolean; }

// ✅ Real use: validated config (after defaults applied)
function startServer(config: Required<Config>) {
  console.log(`Starting on ${config.host}:${config.port}`);
  // All fields guaranteed to exist!
}
Object Types

Readonly<T> — Freeze All Properties

3
Cannot reassign any property
Readonly
interface User { name: string; age: number; }

const user: Readonly<User> = { name: "Alice", age: 25 };

// user.name = "Bob";  // ❌ Error: Cannot assign to 'name'
// user.age = 30;      // ❌ Error: Cannot assign to 'age'

// ✅ Real use: function that should NOT modify input
function displayUser(user: Readonly<User>) {
  console.log(user.name);  // read ✅
  // user.name = "X";      // write ❌ — compile error!
}

// ⚠ WARNING: Readonly is SHALLOW!
interface Deep { data: { value: number } }
const obj: Readonly<Deep> = { data: { value: 1 } };
obj.data.value = 99;  // ✅ This WORKS! Nested is still mutable!
Object Types

Pick<T, K> — Select Properties

4
Keep only the properties you want
Pick
interface User {
  id: number;
  name: string;
  email: string;
  password: string;
  createdAt: Date;
}

// Pick only what a profile card needs:
type ProfileCard = Pick<User, "name" | "email">;
// { name: string; email: string; }

// ✅ Real use: API response — don't expose everything
type PublicUser = Pick<User, "id" | "name" | "email">;
// Excludes password and createdAt
Object Types

Omit<T, K> — Remove Properties

5
Opposite of Pick — exclude properties
Omit
interface User {
  id: number;
  name: string;
  email: string;
  password: string;
}

// Remove sensitive fields:
type SafeUser = Omit<User, "password">;
// { id: number; name: string; email: string; }

// Remove multiple:
type CreateUser = Omit<User, "id" | "password">;
// { name: string; email: string; }
// id is auto-generated, password is hashed separately

// ✅ Pick vs Omit rule of thumb:
// Few to KEEP → use Pick
// Few to REMOVE → use Omit
Object Types

Record<K, V> — Key-Value Map

6
Create an object type from keys and values
Record
// Record<KeyType, ValueType>

// String keys, number values:
const scores: Record<string, number> = {
  math: 90,
  science: 85,
  english: 78,
};

// Union keys — ensures ALL keys exist:
type Status = "active" | "inactive" | "banned";
const statusColors: Record<Status, string> = {
  active: "green",
  inactive: "gray",
  banned: "red",
  // If you miss one, TypeScript will ERROR! ✅
};

// ✅ Real use: lookup tables, dictionaries, configs
type Role = "admin" | "user" | "guest";
const permissions: Record<Role, string[]> = {
  admin: ["read", "write", "delete"],
  user: ["read", "write"],
  guest: ["read"],
};
Union Types

Exclude<T, U> & Extract<T, U>

7
Filter union type members
Unions
type Status = "active" | "inactive" | "banned" | "pending";

// EXCLUDE — remove from union
type ActiveStatuses = Exclude<Status, "banned" | "inactive">;
// "active" | "pending"

// EXTRACT — keep only matching
type BadStatuses = Extract<Status, "banned" | "inactive">;
// "banned" | "inactive"

// Works with any union:
type AllTypes = string | number | boolean | null;
type NonNull = Exclude<AllTypes, null>;        // string | number | boolean
type Primitives = Extract<AllTypes, string | number>;  // string | number

// ⚠ KEY DIFFERENCE from Pick/Omit:
// Pick/Omit → work on OBJECT property keys
// Exclude/Extract → work on UNION members
Function Types

ReturnType<T> — Get Function's Return

8
Extract what a function returns
ReturnType
function getUser() {
  return { id: 1, name: "Alice", role: "admin" as const };
}

// Extract the return type without manually writing it:
type User = ReturnType<typeof getUser>;
// { id: number; name: string; role: "admin" }

// ✅ Real use: keep types in sync with functions
// If getUser() changes its return, User auto-updates!

// Also works with Parameters:
function createUser(name: string, age: number) { }
type CreateParams = Parameters<typeof createUser>;
// [string, number]
Watch Out!

⚠ Common Mistakes

⚠
Mistake 1: Using Partial when you need Pick 
// ❌ Partial makes ALL optional — too loose
function updateName(data: Partial<User>) { }
updateName({});  // ✅ compiles... but name is missing!

// ✅ Pick only what you actually need
function updateName(data: Pick<User, "name">) { }
updateName({});  // ❌ Error: name is required
⚠
Mistake 2: Readonly is SHALLOW 
interface Config { db: { host: string; port: number } }
const cfg: Readonly<Config> = { db: { host: "localhost", port: 5432 } };

// cfg.db = { ... };      // ❌ blocked by Readonly ✅
cfg.db.port = 9999;       // ✅ ALLOWED! Nested is still mutable 😱

// FIX: manually make nested readonly too, or use deep readonly
⚠
Mistake 3: Confusing Omit vs Exclude 
// Omit → removes PROPERTIES from an object type
type NoPassword = Omit<User, "password">;

// Exclude → removes MEMBERS from a union type
type NoNull = Exclude<string | null, null>;  // string

// Don't mix them up!
MistakeWhat happensFix
Partial instead of PickToo permissiveUse Pick for specific fields
Readonly is shallowNested still mutableDeep readonly or spread
Omit vs ExcludeWrong operationOmit=objects, Exclude=unions

Cheatsheet

// OBJECT TRANSFORMATIONS
Partial<User>              // all optional
Required<Config>           // all required
Readonly<Data>             // all readonly
Pick<User, "name"|"email"> // keep specific
Omit<User, "password">     // remove specific
Record<"a"|"b", number>    // { a: number, b: number }

// UNION TRANSFORMATIONS
Exclude<A|B|C, C>          // A | B
Extract<A|B|C, A|B>        // A | B

// FUNCTION TYPES
ReturnType<typeof fn>      // what fn returns
Parameters<typeof fn>      // [arg types]
