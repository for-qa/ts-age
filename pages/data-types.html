Data Types

What kind of data can a variable hold? This session covers all primitive data types in TypeScript — number, string, boolean, null, undefined, any, union, and void — plus the critical difference between type annotation and type inference.

Concepts

Dynamic vs Static Typing

✓

Real-Life Example: Imagine two types of containers:
✓ JavaScript (Dynamic) = A magic box — you put apples in it today, shoes tomorrow, water next week. The box doesn't care what's inside. It changes shape to fit anything.
✓ TypeScript (Static) = A labeled drawer — it says "SOCKS ONLY" on the front. If you try to put a book in the sock drawer, it refuses. Once labeled, only that type goes in.

JS
JavaScript — dynamically typed (anything goes!)
⚠ Flexible
let age = 30;        // age is a number
age = "thirty";      // ✅ No error! Now age is a string
age = true;          // ✅ No error! Now age is a boolean

console.log(age);    // true
// ✓ JavaScript doesn't complain. The type keeps changing.
// This is "dynamic typing" — types are decided at RUNTIME.
TS
TypeScript — statically typed (locked in!)
✅ Safe
let age: number = 30;  // age is a number — locked!
age = "thirty";        // ❌ ERROR: Type 'string' is not assignable to type 'number'
age = true;            // ❌ ERROR: Type 'boolean' is not assignable to type 'number'

// ✓ TypeScript catches the mistake BEFORE your code runs.
// This is "static typing" — types are decided at COMPILE TIME.
FeatureJavaScriptTypeScript
TypingDynamicStatic
Type decided atRuntime (while running)Compile time (before running)
Change type later?AllowedNOT allowed
Catches type bugs?No — finds them at runtimeYes — catches them early

Type Safety — TypeScript's Superpower

✓

Real-Life Example: Think of type safety like a blood type test before a transfusion. You can't give Type A blood to a Type B patient — the hospital checks BEFORE the procedure. JavaScript is like a hospital with no checks — it just gives whatever blood is available and hopes for the best. TypeScript is the careful hospital that verifies everything first.

Type safety means: once a variable has a type, TypeScript makes sure you ONLY store that same type of data in it. It catches mismatches before your code runs.

JS
JavaScript — NOT type safe (silent bugs!)
⚠ Risky
let result = "5" + 3;
console.log(result);  // "53" — NOT 8!

// ✓ "5" is a string, 3 is a number.
// JavaScript silently CONCATENATES them instead of adding.
// No error, no warning — just a wrong answer. This is a HIDDEN BUG.
TS
TypeScript — type safe (catches bugs early!)
✅ Protected
let num1: string = "5";
let num2: number = 3;
let result = num1 + num2;
// ❌ TypeScript warns: are you sure you want to add a string and number?

// ✓ TypeScript catches the problem and tells you about it
// BEFORE the code runs. No surprises!
i

Interview Tip: Dynamic typing = NOT type safe (JavaScript). Static typing = type safe (TypeScript). These two features are connected — one implies the other.

Concepts

Three Key Terms You MUST Know

✓

Real-Life Example:
✓ Type = The LABEL on the drawer (number, string, boolean).
✓ Annotation = YOU write the label yourself with a marker — "This drawer is for socks."
✓ Type Inference = The drawer READS what you put inside and writes its OWN label — "Oh, you put socks in me? I must be a sock drawer!"

Term 1
Type (Data Type)
What
The kind of data a variable holds: number, string, boolean, etc. It's just a label that says what type of stuff goes inside.
Term 2
Annotation
You Tell TypeScript
YOU explicitly write the type using a colon. Example: let age: number = 30. You're the boss — you decide the type.
Term 3
Type Inference
TypeScript Guesses
You DON'T write the type. TypeScript looks at the value and figures it out itself. Example: let age = 30 — TS knows it's a number.
1
Annotation — you specify the type explicitly
You → TS
let age: number = 30;      // YOU said: "this is a number"
let name: string = "John"; // YOU said: "this is a string"
let active: boolean = true; // YOU said: "this is a boolean"

// The colon + type is the ANNOTATION
//       ✓ this part
// let age: number = 30;
//        ^^^^^^^^
2
Type Inference — TypeScript figures it out
TS → Auto
let age = 30;       // TypeScript says: "30 is a number, so age = number"
let name = "John";  // TypeScript says: "John is text, so name = string"
let active = true;  // TypeScript says: "true is boolean, so active = boolean"

// No colon, no type written — TypeScript INFERS it from the value!
// Hover over the variable in VS Code to see what type TS assigned.
AspectAnnotationType Inference
Who decides the type?You (the developer)TypeScript (automatic)
Syntaxlet age: number = 30let age = 30
Colon used?Yes — : typeNo colon needed
When to use?Recommended — clearer codeOkay for simple, obvious cases
⚠

Interview Question: "What is the difference between annotation and type inference?" — Annotation = you EXPLICITLY set the type. Inference = TypeScript AUTOMATICALLY detects the type from the value. They're opposites!

Primitive Types

number — Integers & Decimals

✓

Real-Life Example: Think of number as a calculator display. It shows whole numbers (42) and decimal numbers (3.14) — there's no separate button for "integer" or "decimal." It's all just numbers. Same in TypeScript — one type handles both!

Unlike Java (which has int, float, double, long), TypeScript has just ONE type for all numbers: number. It works for integers, decimals, and big numbers alike.

1
number examples — integers & decimals
One type for all
let age: number = 30;               // integer (whole number)
let price: number = 25.5;           // decimal (floating point)
let bigNumber: number = 9999999;    // large number — no limit!

console.log("Age:", age);           // Age: 30
console.log("Price:", price);       // Price: 25.5
console.log("Big:", bigNumber);     // Big: 9999999
console.log(typeof age);            // "number"

typeof is a built-in method that returns the data type of any variable as a string. Works for all types.

i

Console Tip: To print a label with a value, use a comma: console.log("Age:", age) — the string prints as-is, followed by the variable's value.

Primitive Types

string — Text Data

✓

Real-Life Example: A string is like a text message on your phone. It can contain letters, numbers, emojis, spaces — anything you type. Whether it's "Hello" or "Room 404" or "✓ Party!", it's all just text. And just like messages, you can write them in three different ways — like texting, emailing, or handwriting a letter.

Strings represent text — any sequence of characters. You can create them in three ways:

1
Double quotes — the standard way
"text"
let firstName: string = "John";
console.log(firstName);  // John
2
Single quotes — also valid
'text'
let lastName: string = 'Kennedy';
console.log(lastName);  // Kennedy
3
Backticks — for combining text + variables (template literals)
`text ${var}`
let firstName: string = "John";
let lastName: string = "Kennedy";

// Backtick lets you INSERT variables inside the string!
let greeting: string = `Hello ${firstName} ${lastName}`;

console.log(greeting);  // Hello John Kennedy

// Syntax: use backtick ` (not single quote ')
// Put variables inside: ${variableName}
// Dollar sign + curly braces = "give me the VALUE of this variable"

Backticks are used when you want to mix text with variables. This is called a template literal. Instead of messy concatenation, you just drop variables right into the text!

StyleSyntaxCan insert variables?When to use
Double quotes"Hello"NoSimple strings
Single quotes'Hello'NoSimple strings
Backticks`Hello ${name}`Yes!Dynamic strings with variables
⚠

Common Mistake: Using double quotes with ${} won't work: "Hello ${name}" prints literally Hello ${name}. You MUST use backticks ` for template literals.

Primitive Types

boolean — True or False

✓

Real-Life Example: A boolean is like a light switch — it's either ON (true) or OFF (false). There's no "maybe." No "half on." Just two options: yes or no, on or off, true or false.

Boolean variables hold only two possible values: true or false. They're used for decisions — is the user logged in? Is the score passing? Is it raining?

1
boolean examples — yes/no questions
true / false
let isStudent: boolean = true;
let hasJob: boolean = false;
let isLoggedIn: boolean = true;

console.log("Is Student:", isStudent);  // Is Student: true
console.log("Has Job:", hasJob);        // Has Job: false
console.log(typeof isStudent);          // "boolean"

Common naming convention: start boolean variable names with is, has, can, or should — like isActive, hasPermission, canEdit.

Primitive Types

null & undefined — Empty Values

✓

Real-Life Example:
✓ null = An empty mailbox — the mailbox exists, but there's nothing inside. You KNOW it's empty.
❝ undefined = A mailbox that was never built — there's no mailbox at all. Nobody set it up yet.

Both mean "no value," but they're slightly different:

1
null — intentionally empty ("I know there's nothing")
Known empty
let emptyValue: null = null;
console.log(emptyValue);      // null
console.log(typeof emptyValue); // "object" (a known JavaScript quirk!)

// null means: "I deliberately set this to nothing."
// emptyValue = 10;  ← ❌ ERROR: can only hold null
2
undefined — never assigned ("Nobody put anything here")
Not set
let notAssigned: undefined = undefined;
console.log(notAssigned);     // undefined

// More commonly, undefined appears when you FORGET to assign a value:
let price: number;
console.log(price);  // undefined — no value was assigned yet!
// TypeScript warns: "Variable 'price' is used before being assigned"
Aspectnullundefined
MeaningIntentionally emptyNever assigned / not set up
Who sets it?You (the developer)TypeScript (automatically)
Common usage"Reset" a variable to nothingForgot to set a value
i

In practice, you rarely declare variables as null or undefined types. These values are mostly returned by TypeScript when something is missing or not set.

Primitive Types

any — The "I Don't Care" Type

✓

Real-Life Example: any is like a junk drawer in your kitchen. You can put ANYTHING in there — scissors, batteries, tape, candy, keys. There are no rules. It's convenient but messy — and you'll never find what you need quickly. That's why organized people (and good code) avoid junk drawers!

any is a special type that turns off TypeScript's type checking. The variable accepts ANY type of data — numbers, strings, booleans, anything. It's like going back to JavaScript mode.

!
any — accepts everything, checks nothing
⚠ Use Carefully
let value: any = "Hello";     // string — OK
console.log(typeof value);    // "string"

value = 42;                   // number — also OK!
console.log(typeof value);    // "number"

value = true;                 // boolean — still OK!
console.log(typeof value);    // "boolean"

// ✓ No errors at all! any turns off ALL type checking.
// This BREAKS type safety — use only when absolutely necessary.
⚠

Why avoid any? It defeats the purpose of TypeScript. You lose: (1) static typing, (2) type safety, (3) autocompletion in your editor. It's like buying a smartphone and only using it as a paperweight.

Interview Question: "Can you break TypeScript's type safety?" — Yes, using the any type. But it's not recommended because it loses TypeScript's core benefits.

Primitive Types

Union Type — "This OR That"

✓

Real-Life Example: A union type is like a parking spot that accepts both cars AND bikes — but ONLY cars and bikes. Not trucks, not boats. You specify exactly which types are allowed. It's like any but with rules.

A union type lets a variable hold multiple specific types — but only the ones YOU list. It's the sweet spot between strict single-type and chaotic any.

|
Union — combine types with the pipe | operator
Controlled flexibility
// ID can be a number OR a string — nothing else
let userId: number | string | boolean;

userId = "ABC123";     // ✅ string is allowed
console.log(userId);   // ABC123

userId = 12345;        // ✅ number is allowed
console.log(userId);   // 12345

userId = true;         // ✅ boolean is allowed
console.log(userId);   // true

userId = [1, 2, 3];   // ❌ ERROR: array is NOT in the union!
// Only number, string, boolean are allowed — you listed them.
ApproachSyntaxAllowed typesType safe?
Single typelet x: numberOnly numberYes
Union typelet x: number | stringNumber or stringYes
anylet x: anyEverythingNo
i

Union vs Any: Union is like ordering "coffee OR tea" — you get a choice, but it's limited. any is like ordering "whatever you have" — no control at all. Prefer union when you know the options!

Primitive Types

void — "Returns Nothing"

✓

Real-Life Example: void is like a vending machine that takes your money but gives you nothing back — it DOES something (plays a sound, shows a message), but it doesn't return any product. The function does work but doesn't give back a result.

void is used specifically with functions — not variables. It means: "this function does something but doesn't return a value."

1
void — function that prints but returns nothing
No return
function showWelcome(): void {
  console.log("Welcome to TypeScript!");
  // ✓ It PRINTS something, but doesn't RETURN anything
}

showWelcome();  // Welcome to TypeScript!
2
Function WITH a return type (not void)
Returns value
function addNumbers(x: number, y: number): number {
  return x + y;  // Returns a NUMBER
}

let result: number = addNumbers(10, 20);
console.log(result);  // 30
Function typeReturn typeUses return?Example
Prints onlyvoidNoconsole.log("Hi")
Returns a valuenumber, string, etc.Yesreturn x + y
i

void is optional in TypeScript — if you skip it, TypeScript infers the return type. But writing it explicitly makes your code clearer: "I intentionally made this function return nothing."

Watch Out!

⚠ Common Beginner Mistakes

⚠
Mistake 1: typeof null returns "object" (not "null"!)

This is a famous JavaScript bug from 1995 that was never fixed. typeof null says "object" instead of "null". If you use typeof to check for null, you'll get the WRONG answer!

// ❌ WRONG — typeof null is "object", not "null"!
let data = null;
console.log(typeof data); // "object" ← SURPRISE!

if (typeof data === "null") {
  console.log("It's null!"); // NEVER runs!
}

// ✅ CORRECT — use === null to check for null
if (data === null) {
  console.log("It's null!"); // ✓ Works!
}

Rule: Never use typeof to check for null. Always use === null directly.

⚠
Mistake 2: Number + String = String (not a number!)

When you use + with a number and a string, JavaScript converts the number to a string and GLUES them together. This is called type coercion and it catches every beginner.

let price: number = 100;
let tax: string = "20";

// ❌ WRONG — string + number = concatenation!
let total = price + tax;
console.log(total);       // "10020" (not 120!)
console.log(typeof total); // "string" ← it became a string!

// ✅ CORRECT — convert string to number first
let totalFixed = price + Number(tax);
console.log(totalFixed);       // 120 ✓
console.log(typeof totalFixed); // "number" ✓

Rule: Always check your data types before doing math. Use Number() or parseInt() to convert strings to numbers.

⚠
Mistake 3: Using any defeats the whole purpose of TypeScript!

When you type any, TypeScript stops checking that variable completely. No autocomplete, no error warnings, no safety. It's like wearing a seatbelt but not buckling it.

// ❌ BAD — any removes all type safety
let data: any = "hello";
data = 42;          // No error (but is this intentional?)
data.toUpperCase(); // No error in editor, CRASHES at runtime!
// TypeError: data.toUpperCase is not a function

// ✅ BETTER — use union types for flexibility WITH safety
let data2: string | number = "hello";
data2 = 42;            // ✓ Allowed
data2.toUpperCase();   // ❌ TypeScript catches this! Error at compile time

Rule: Avoid any whenever possible. If a variable can hold multiple types, use a union type (string | number) instead — you get flexibility AND safety.

Key terms

// Annotation — YOU write the type
let score: number = 100;  // ✓ ": number" is the annotation

// Type Inference — TypeScript DETECTS the type
let score = 100;          // ✓ TS knows it's a number automatically

// typeof — check the type at runtime
console.log(typeof score);  // "number"

Dynamic vs Static typing

// JavaScript (Dynamic) — type can change anytime
let x = 10;       // number
x = "hello";      // ✅ now it's a string — allowed!

// TypeScript (Static) — type is LOCKED
let y: number = 10;  // number
y = "hello";         // ❌ ERROR — type can't change!
