What are Design Patterns?

✓

Real-Life: Design patterns are like recipes 📖. You don't invent a new way to make roti every time — you follow a proven recipe. Design patterns are proven recipes for solving common coding problems.

Patterns

Singleton — Only One Instance

✓

Analogy: There's only ONE President of India at a time. No matter how many times you ask "who's the president?", you always get the same person. That's a Singleton.

1
Ensure only one instance exists
Singleton 
class Database {
  private static instance: Database;
  private connectionCount = 0;

  // Private constructor — can't use 'new Database()' outside!
  private constructor() {
    console.log("Database connection created");
  }

  // Only way to get the instance
  static getInstance(): Database {
    if (!Database.instance) {
      Database.instance = new Database();
    }
    return Database.instance;
  }

  query(sql: string) {
    this.connectionCount++;
    console.log(`[Query #${this.connectionCount}] ${sql}`);
  }
}

// Usage:
const db1 = Database.getInstance();
const db2 = Database.getInstance();
console.log(db1 === db2);  // true — same instance!

db1.query("SELECT * FROM users");
db2.query("INSERT INTO logs");  // uses same connection

// new Database();  // ❌ Error: constructor is private

// ✅ Use when: database connections, config managers, loggers
// ❌ Avoid when: unit testing (hard to mock/reset)

Factory — Create Without Knowing the Class

✓

Analogy: You go to a car dealership and say "I want an SUV." You don't build the car yourself — the factory builds the right car based on your request.

2
Delegate object creation to a factory
Factory 
interface Notification {
  send(message: string): void;
}

class EmailNotification implements Notification {
  send(message: string) { console.log(`📧 Email: ${message}`); }
}
class SMSNotification implements Notification {
  send(message: string) { console.log(`📱 SMS: ${message}`); }
}
class PushNotification implements Notification {
  send(message: string) { console.log(`🔔 Push: ${message}`); }
}

// The Factory — creates the right type based on input
type NotifType = "email" | "sms" | "push";

function createNotification(type: NotifType): Notification {
  switch (type) {
    case "email": return new EmailNotification();
    case "sms":   return new SMSNotification();
    case "push":  return new PushNotification();
  }
}

// Usage — caller doesn't know which class is created
const notif = createNotification("email");
notif.send("Hello!");  // 📧 Email: Hello!

// ✅ Use when: creating objects based on config/input
// ✅ Adding new types = just add a new case

Observer — Subscribe to Events

✓

Analogy: Like subscribing to a YouTube channel. When a new video (event) is uploaded, ALL subscribers get notified automatically. You don't check manually — the notification comes to you.

3
One-to-many event notification
Observer 
type Listener<T> = (data: T) => void;

class EventEmitter<T> {
  private listeners: Listener<T>[] = [];

  subscribe(listener: Listener<T>): () => void {
    this.listeners.push(listener);
    // Return unsubscribe function
    return () => {
      this.listeners = this.listeners.filter(l => l !== listener);
    };
  }

  emit(data: T): void {
    this.listeners.forEach(listener => listener(data));
  }
}

// Usage:
interface CartEvent { item: string; price: number; }
const cartEvents = new EventEmitter<CartEvent>();

// Subscribe multiple listeners
const unsub1 = cartEvents.subscribe(e =>
  console.log(`🛒 Added: ${e.item}`)
);
const unsub2 = cartEvents.subscribe(e =>
  console.log(`💰 Total +₹${e.price}`)
);

cartEvents.emit({ item: "Laptop", price: 50000 });
// 🛒 Added: Laptop
// 💰 Total +₹50000

unsub1();  // unsubscribe first listener
cartEvents.emit({ item: "Mouse", price: 500 });
// 💰 Total +₹500  (only second listener fires)

Strategy — Swappable Algorithms

✓

Analogy: Like choosing a payment method at checkout — UPI, credit card, or cash. The checkout process is the same, but the payment strategy is swappable.

4
Swap behavior at runtime
Strategy 
// Define strategy interface
interface SortStrategy<T> {
  sort(data: T[]): T[];
}

// Concrete strategies
class BubbleSort<T> implements SortStrategy<T> {
  sort(data: T[]): T[] {
    console.log("Using bubble sort (simple, slow)");
    return [...data].sort();
  }
}
class QuickSort<T> implements SortStrategy<T> {
  sort(data: T[]): T[] {
    console.log("Using quick sort (fast)");
    return [...data].sort();
  }
}

// Context that uses the strategy
class Sorter<T> {
  constructor(private strategy: SortStrategy<T>) {}

  setStrategy(strategy: SortStrategy<T>) {
    this.strategy = strategy;
  }

  sort(data: T[]): T[] {
    return this.strategy.sort(data);
  }
}

// Swap strategies at runtime!
const sorter = new Sorter(new BubbleSort<number>());
sorter.sort([3, 1, 2]);  // "Using bubble sort"

sorter.setStrategy(new QuickSort());
sorter.sort([3, 1, 2]);  // "Using quick sort"
Watch Out!

⚠ Common Mistakes

⚠
Mistake 1: Overusing patterns

Not every problem needs a pattern. A simple function is often better than a Factory class. Use patterns when complexity justifies them. 

// ❌ Overkill for simple cases
class GreeterFactory {
  create(name: string) { return `Hello ${name}`; }
}

// ✅ Just use a function
const greet = (name: string) => `Hello ${name}`;
⚠
Mistake 2: Singleton makes testing hard 
// Singletons carry state between tests!
// FIX: Allow reset in test environments, or use dependency injection
Reference

When to Use Which?

Pattern Use when Real examples
Singleton Need exactly ONE instance DB connection, logger, config
Factory Create objects based on input Notifications, UI components
Observer React to events/changes Event bus, state management
Strategy Swap algorithms at runtime Sorting, payment, auth methods
