Scope

3 Types of Scope

🌍
Global
Visible everywhere
🏠
Function
Visible only inside function
📦
Block
Visible only inside { }
1
Where can a variable be seen?
Scope 
// GLOBAL scope — visible everywhere
const globalVar = "I'm global";

function demo() {
  // FUNCTION scope — only visible inside this function
  const funcVar = "I'm local to demo()";

  if (true) {
    // BLOCK scope — only visible inside this { }
    const blockVar = "I'm local to this block";
    let blockLet = "me too";
    var funcScoped = "BUT var is function-scoped!";

    console.log(globalVar);   // ✅ accessible
    console.log(funcVar);     // ✅ accessible
    console.log(blockVar);    // ✅ accessible
  }

  console.log(funcScoped);    // ✅ var leaks out of block!
  // console.log(blockVar);   // ❌ ReferenceError
  // console.log(blockLet);   // ❌ ReferenceError
}

// console.log(funcVar);      // ❌ ReferenceError

Key: let/const = block-scoped. var = function-scoped (leaks out of blocks!).

Scope

Lexical Scope

✓

Real-Life: Lexical scope is like family inheritance. A child has access to everything in their parent's house, the grandparent's house, and so on — but the parent can't access the child's room.

2
Functions look UP the chain, never down
Lexical 
const grandparent = "👵 Grandparent";

function parent() {
  const parentVar = "👩 Parent";

  function child() {
    const childVar = "👧 Child";

    // Child can see UP the chain:
    console.log(childVar);    // ✅ own
    console.log(parentVar);   // ✅ parent's
    console.log(grandparent); // ✅ grandparent's
  }

  child();
  // console.log(childVar);  // ❌ parent can't see child's!
}

// Scope is determined by WHERE the function is WRITTEN,
// NOT where it is called. This is "lexical" scoping.
Closures

What is a Closure?

✓

Real-Life: A closure is like a backpack 🎒. When a function "graduates" from its school (scope), it carries a backpack with all the variables it still needs. Even after the school is gone, the function still has its backpack.

!
A function + its remembered variables
Core Concept 
function createGreeter(greeting: string) {
  // 'greeting' lives here

  return function(name: string) {
    // This inner function REMEMBERS 'greeting'
    // even after createGreeter() has finished running!
    console.log(`${greeting}, ${name}!`);
  };
}

const sayHello = createGreeter("Hello");
const sayNamaste = createGreeter("Namaste");

// createGreeter() has FINISHED executing...
// but the returned functions still remember their 'greeting'!

sayHello("Alice");    // "Hello, Alice!"
sayNamaste("Bob");    // "Namaste, Bob!"

// THAT is a closure:
// The inner function CLOSED OVER the variable 'greeting'
// and remembers it forever
Closures

Classic Pattern: Counter

3
Private state with closures
Counter 
function createCounter() {
  let count = 0;  // private! not accessible from outside

  return {
    increment: () => { count++; },
    decrement: () => { count--; },
    getCount: () => count,
  };
}

const counter = createCounter();
counter.increment();
counter.increment();
counter.increment();
counter.decrement();
console.log(counter.getCount());  // 2

// count is PRIVATE — no one can access it directly
// counter.count  → undefined (doesn't exist on the object)

// Each counter is independent:
const counter2 = createCounter();
counter2.increment();
console.log(counter2.getCount());  // 1 (separate from counter)
Closures

Private Variables (Module Pattern)

4
Encapsulation before classes existed
Privacy 
function createBankAccount(initialBalance: number) {
  let balance = initialBalance;  // PRIVATE — closure variable
  const transactions: string[] = [];

  return {
    deposit(amount: number) {
      balance += amount;
      transactions.push(`+${amount}`);
    },
    withdraw(amount: number) {
      if (amount > balance) {
        console.log("Insufficient funds!");
        return;
      }
      balance -= amount;
      transactions.push(`-${amount}`);
    },
    getBalance: () => balance,
    getHistory: () => [...transactions],  // return copy!
  };
}

const account = createBankAccount(1000);
account.deposit(500);
account.withdraw(200);
console.log(account.getBalance());  // 1300
// account.balance → undefined (PRIVATE!)
Closures

Factory Functions

5
Generate customized functions
Factory 
// Multiplier factory
function createMultiplier(factor: number) {
  return (num: number) => num * factor;
}

const double = createMultiplier(2);
const triple = createMultiplier(3);
const toINR = createMultiplier(83.5);

console.log(double(5));   // 10
console.log(triple(5));   // 15
console.log(toINR(100));  // 8350

// Logger factory
function createLogger(prefix: string) {
  return (msg: string) => console.log(`[${prefix}] ${msg}`);
}

const info = createLogger("INFO");
const error = createLogger("ERROR");

info("Server started");   // [INFO] Server started
error("Disk full");        // [ERROR] Disk full
Classic Traps

The Classic: var in Loops

⚠

This is the #1 closure interview question. You WILL be asked this.

!
var creates SHARED closure — let creates separate ones
Interview Classic 
// ❌ THE TRAP — var is function-scoped, NOT block-scoped
for (var i = 0; i < 3; i++) {
  setTimeout(() => {
    console.log(i);  // What prints?
  }, 100);
}
// Prints: 3, 3, 3  😱 (NOT 0, 1, 2!)
// WHY? All 3 callbacks share the SAME 'i' (var = function scope)
// By the time setTimeout fires, i is already 3

// ✅ FIX 1 — use 'let' (block-scoped = separate closure each iteration)
for (let i = 0; i < 3; i++) {
  setTimeout(() => {
    console.log(i);  // 0, 1, 2 ✅
  }, 100);
}

// ✅ FIX 2 — IIFE (old school, creates manual closure)
for (var i = 0; i < 3; i++) {
  ((j) => {
    setTimeout(() => {
      console.log(j);  // 0, 1, 2 ✅
    }, 100);
  })(i);
}

Memory Considerations

!
Closures keep variables alive in memory
Memory 
// Closures prevent garbage collection of closed-over variables

function createHeavyClosure() {
  const hugeArray = new Array(1_000_000).fill("data"); // 1M items

  return () => {
    console.log(hugeArray.length);  // hugeArray stays in memory!
  };
}

const fn = createHeavyClosure();
// hugeArray is NOT garbage collected — fn still references it!

// ✅ FIX — only close over what you need
function createLightClosure() {
  const hugeArray = new Array(1_000_000).fill("data");
  const length = hugeArray.length;  // extract what you need

  return () => {
    console.log(length);  // only 'length' is kept, not hugeArray
  };
}
// hugeArray CAN be garbage collected now
Watch Out!

⚠ Common Mistakes

⚠
Mistake 1: var in loops creates shared closure 
// All callbacks see the FINAL value of var
for (var i = 0; i < 5; i++) {
  setTimeout(() => console.log(i), 100);
}
// 5, 5, 5, 5, 5 — NOT 0, 1, 2, 3, 4
// FIX: use let instead of var
⚠
Mistake 2: Accidentally keeping large objects alive 
// If you close over a huge object but only need one field,
// extract the field first. Don't capture the whole object.
// BAD: return () => bigObj.name;
// GOOD: const name = bigObj.name; return () => name;
⚠
Mistake 3: Thinking closures are complicated 
// A closure is JUST a function that remembers variables
// from where it was CREATED.

// You use closures ALL THE TIME without realizing:
const name = "Alice";
button.addEventListener("click", () => {
  console.log(name);  // This IS a closure!
});
Mistake What happens Fix
var in loop All callbacks share same value Use let
Large objects in closure Memory leak Extract needed values
Overthinking closures Confusion Function + remembered vars

Cheatsheet

// SCOPE: where a variable is visible
// Global > Function > Block (let/const)
// var = function-scoped, let/const = block-scoped

// LEXICAL SCOPE: determined by WHERE function is written
// Inner functions can see outer variables, not vice versa

// CLOSURE: function + its remembered outer variables
function outer() {
  const x = 10;
  return () => x;  // closure over x
}

// PATTERNS:
// Counter:    return { increment, getCount }
// Private:    let private = ...; return { getters/setters }
// Factory:    return customized function
// var trap:   use let in loops, not var
