Core Concept

What is an Interface?

✓

Real-Life Example: An interface is like a job application form template. It says "fill in: Name, Age, Email" -- it defines the STRUCTURE (what fields exist), but it doesn't fill in any data or do any work. A class is like the actual employee who fills in the form and starts working.

An interface defines the structure of an object -- what properties and methods it should have. It's a blueprint, just like a class, but with key differences:

Feature Class Interface
Properties ✅ ✅
Constructor ✅ ❌
Methods with body ✅ ❌ (abstract only)
Create objects ✅ new ✅ (no new)
Access modifiers ✅ ❌ 
// Syntax
interface InterfaceName {
  property1: type;
  property2: type;
  methodName(): returnType;  // abstract  --  signature only, no body!
}
Example 1

Basic Interface -- Creating Objects

interface Person {
  name: string;
  age: number;
}

// Create object  --  NO 'new' keyword needed!
let student: Person = {
  name: "John",
  age: 25
};

console.log(student.name);  // "John"
console.log(student.age);   // 25
console.log(student);       // { name: "John", age: 25 }
⚠

No new keyword! Unlike classes, interfaces cannot be instantiated with new. You create objects by directly assigning data that matches the structure.

Example 2

Optional Properties -- Value Not Required

✓

Real-Life Example: Like a registration form where "Department" is optional. Some employees have one, some don't. If left blank → undefined. 

interface Employee {
  eid: number;
  ename: string;
  edepartment?: string;  // ← optional (question mark)
}

// Object 1  --  skip optional field
let emp: Employee = { eid: 101, ename: "John" };
console.log(emp.edepartment);  // undefined

// Object 2  --  include optional field
let emp1: Employee = {
  eid: 102,
  ename: "Smith",
  edepartment: "Accounts"
};
console.log(emp1.edepartment);  // "Accounts"
Example 3

Readonly Properties -- Assign Once, Never Change

✓

Real-Life Example: A book's ISBN number -- once printed, it can NEVER be changed. You can read it, but you can't modify it. 

interface Book {
  title: string;
  readonly isbn: string;  // ← readonly keyword
}

let b1: Book = {
  title: "Learn Playwright",
  isbn: "123ABC"
};

// ✅ Regular property  --  can be changed
b1.title = "Learn TypeScript";
console.log(b1.title);  // "Learn TypeScript"

// ❌ Readonly  --  cannot change!
b1.isbn = "123XYZ";
// ERROR: Cannot assign to 'isbn' because it is a read-only property
Methods

Abstract Methods -- Signature Only, No Body

✓

Real-Life Example: An abstract method is like a blank check. The check template says "Amount: ___" (signature), but the actual value (implementation) is filled in later by whoever uses it.

Interface methods are abstract by default -- they have only a signature (name + params + return type). The body is implemented when creating the object or in a class. 

interface Book {
  title: string;
  readonly isbn: string;
  display(): void;  // ← abstract method (no body!)
}

let b1: Book = {
  title: "Learn Playwright",
  isbn: "123ABC",

  // Implement the abstract method HERE
  display(): void {
    console.log(b1.title + " | " + b1.isbn);
  }
};

b1.display();  // "Learn Playwright | 123ABC"
i

No abstract keyword needed! In interfaces, methods are abstract by default. In abstract classes, you must use the abstract keyword explicitly.

⚠

No this keyword! Inside interface object methods, you cannot use this to access properties. Use the object name directly: b1.title instead of this.title. The this keyword only works inside classes.

Inheritance

Extending Interfaces -- Parent & Child

✓

Real-Life Example: A generic "Animal" form has a name field. A "Dog" form extends it by adding a color field. The dog form inherits the name field AND adds its own.

Just like classes, interfaces can extend other interfaces using extends. 

// Parent interface
interface Animal {
  name: string;
}

// Child interface  --  inherits 'name', adds 'color'
interface Dog extends Animal {
  color: string;
}

// Object uses BOTH properties
let myDog: Dog = {
  name: "Buddy",   // from Animal
  color: "Black"   // from Dog
};

console.log(myDog.name);   // "Buddy"
console.log(myDog.color);  // "Black"
i

Multiple inheritance! Unlike classes (single inheritance only), interfaces support extending MULTIPLE parent interfaces: interface C extends A, B { }

Inheritance

Class implements Interface

✓

Real-Life Example: An interface is like a contract: "You MUST have a name and MUST implement the sound() method." A class that implements it says: "OK, I'll fulfill the contract and provide the actual code."

A class can implement an interface. This means the class MUST:

  1. Re-declare all interface properties
  2. Implement all abstract methods
☑
Interface → Class implementation
implements 
interface Animal {
  name: string;
  sound(): void;   // abstract method
}

class Dog implements Animal {
  name: string;      // MUST re-declare!
  color: string;     // additional property (optional)

  constructor(name: string, color: string) {
    this.name = name;
    this.color = color;
  }

  // MUST implement abstract method!
  sound(): void {
    console.log("Bark!");
  }
}

let pet = new Dog("Buddy", "Black");
console.log(pet.name);   // "Buddy"
console.log(pet.color);  // "Black"
pet.sound();              // "Bark!"

The class can add extra properties (like color) and a constructor -- things the interface cannot have.

When implementing an interface Required?
Re-declare ALL interface properties in the class ✅ Mandatory
Implement ALL abstract methods ✅ Mandatory
Add a constructor Optional (but recommended)
Add extra properties/methods Optional
⚠

extends vs implements: Class extends Class (properties auto-inherited). Class implements Interface (properties must be re-declared). Don't mix them up!

Theory

Normal Class vs Abstract Class vs Interface

Feature Normal Class Abstract Class Interface
Properties ✅ ✅ ✅
Normal methods (with body) ✅ ✅ ❌
Abstract methods (no body) ❌ ✅ (needs abstract keyword) ✅ (default)
Constructor ✅ ✅ ❌
Instantiation (new) ✅ ❌ ❌ (direct)

Interface is the most "strict" -- only abstract methods, no implementation. Abstract class is in between. Normal class requires full implementation.

Theory

Inheritance Rules -- What Can Extend/Implement What?

Combination Keyword Possible?
Class → Class extends ✅ (single only)
Interface → Interface extends ✅ (multiple OK)
Class → Interface implements ✅ (multiple OK)
Interface → Class -- ❌ Not possible 
// ✅ Class extends Class (single)
class Honda extends Car { }

// ✅ Interface extends Interface (multiple OK)
interface Dog extends Animal, Pet { }

// ✅ Class implements Interface (multiple OK)
class Dog implements Animal, Pet { }

// ❌ Interface cannot implement/extend a Class
// Why? Class has implementation  --  interface doesn't allow it
Comparison

Class vs Interface -- Full Comparison

Feature Class Interface
Definition Blueprint with implementation Structure/type with NO implementation
Contains Properties + Constructor + Methods Properties + Abstract methods only
Instantiation ✅ new ClassName() ❌ Cannot use new
Inheritance Single only (extends) Multiple OK (extends)
Implementation Can implement interfaces Cannot implement classes
Access Modifiers ✅ public, protected, private ❌ Not supported
Static ✅ Supported ❌ Not supported
Constructor ✅ Supported ❌ Not supported
this keyword ✅ Works ❌ Not inside object
Watch Out!

⚠ Common Beginner Mistakes

⚠
Mistake 1: Trying to create an object with new Interface()

An interface is just a blueprint on paper -- it describes a shape but has NO actual code inside. You cannot use new on it. Only classes can be instantiated with new. 

interface Animal {
  name: string;
  sound(): string;
}

// ❌ WRONG  --  cannot instantiate an interface!
let dog = new Animal();  // Error: 'Animal' only refers to a type

// ✅ CORRECT  --  create a class that implements the interface
class Dog implements Animal {
  name: string;
  constructor(name: string) { this.name = name; }
  sound() { return "Woof!"; }
}
let dog = new Dog("Buddy"); // ✓ Works!

Rule: Interfaces = blueprint (no new). Classes = factory (use new). Think: you can't build a house from just the architectural drawing -- you need actual bricks (class)!

⚠
Mistake 2: Forgetting to implement ALL required properties

When a class implements an interface, it MUST provide ALL required properties and methods. Missing even one gives a compile error! 

interface Vehicle {
  brand: string;
  speed: number;
  drive(): void;
}

// ❌ WRONG  --  missing 'speed' and 'drive()'
class Car implements Vehicle {
  brand = "Toyota";
  // Error: Class 'Car' incorrectly implements interface 'Vehicle'
  // Property 'speed' is missing
  // Property 'drive' is missing
}

// ✅ CORRECT  --  all properties and methods implemented
class CarFixed implements Vehicle {
  brand = "Toyota";
  speed = 120;
  drive() { console.log("Driving!"); }
}

Rule: Check the interface like a checklist -- every non-optional property must be in your class. TypeScript will tell you exactly which ones are missing!

Mistake What you wrote What happens Fix
new Interface() new Animal() Compile error Create a class, then new Dog()
Missing property Class missing required field Compile error Implement ALL interface properties

Index Signatures -- Dynamic Keys

✓

Real-Life: An index signature is like a dictionary -- you don't know all the words in advance, but you know every entry has a word (string key) and a definition (value of a specific type).

When you don't know all property names ahead of time but know their shape, use an index signature:

I
Dynamic keys with type safety
Index Signatures 
// Index signature  --  any string key, number value
interface ScoreBoard {
  [playerName: string]: number;
}

const scores: ScoreBoard = {
  Alice: 95,
  Bob: 87,
  Charlie: 92,
};
scores["Dave"] = 78;  // ✅ dynamic key works!

// Mix fixed + dynamic keys
interface ApiResponse {
  status: number;          // fixed key  --  always present
  message: string;         // fixed key  --  always present
  [key: string]: unknown;  // any other key is allowed
}

const res: ApiResponse = {
  status: 200,
  message: "OK",
  data: [1, 2, 3],     // ✅ allowed by index signature
  timestamp: Date.now() // ✅ allowed by index signature
};

// Readonly index signature  --  can't add/modify after creation
interface ReadonlyMap {
  readonly [key: string]: string;
}
const config: ReadonlyMap = { theme: "dark", lang: "en" };
// config.theme = "light";  // ❌ Error  --  readonly!

Rule: The index signature value type must be compatible with ALL fixed property types. If [key: string]: number, then every property must be number.

Quick Reference Cheatsheet

Define an interface

interface Person {
  name: string;           // regular
  email?: string;         // optional
  readonly id: number;    // readonly
  greet(): void;          // abstract method
}

Create object (no new keyword)

let p: Person = {
  name: "John",
  id: 101,
  greet() { console.log("Hi!"); }
};

Extend interface

interface Dog extends Animal { color: string; }

Implement interface with class

class Dog implements Animal {
  name: string;  // must re-declare!
  constructor(name: string) { this.name = name; }
  sound() { console.log("Bark!"); }  // must implement!
}

Key rules

// ✅ Interface = structure only, no implementation
// ✅ Methods are abstract by default (no 'abstract' keyword needed)
// ✅ No constructor, no access modifiers, no static
// ✅ Cannot use 'new' to create objects
// ✅ Cannot use 'this' inside interface objects
// ✅ Interface extends Interface = OK (multiple OK)
// ✅ Class implements Interface = OK (must re-declare + implement)
// ❌ Interface cannot implement/extend a Class
